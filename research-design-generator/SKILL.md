# Skill: research-design-generator

粗い研究着想を、標準問題、失敗機構、提案機構、観測予測、比較実験、評価指標、論文貢献へ変換する。目的は発想支援ではなく、査読可能な因果主張と検証設計への圧縮である。

## 原則

- 研究設計を「応用場面 -> 標準問題 -> 失敗モード -> 提案機構 -> 仮説 -> 観測予測 -> 比較条件 -> 指標 -> 貢献」の変換列として扱う。
- 人工生命、複雑系、マルチエージェント、行動生態、認知科学、分散制御の着想では、対象現象の再現ではなく `局所 cue -> target selection / action selection -> 局所行動 -> 構造・環境記憶 -> 物理場・情報場 -> 目的機能` の閉ループ因果列へ写像する。
- 研究の入口は、一般的な重要性ではなく「通常なら全体地図、中央制御、目的対象の直接観測が必要に見えるが、それなしに成立するのか」という逆説から置く。
- 具体語を中核手順へ固定しない。経験共有、オンライン更新、媒介変数、役割形成は条件付き拡張として扱う。
- 中心仮説は観測可能な内部変化、構造変化、性能変化、副作用へ分解する。
- 主体が観測できる cue と観測できない情報を分離する。目的対象、全体地図、最終形状、グローバル目的関数を観測していない前提なら、手法や評価へ暗黙に入れない。
- 環境設定とエージェント設定を同じ重みで設計する。境界条件、初期構造、外乱、目的領域、環境アブレーションを、行動則や cue 条件と同格の実験要因として扱う。
- action selection は「何をするか」、target selection は「どこで、何を対象にするか」として分ける。局所 cue がどちらを制御するのかを明記する。
- 各主張に、対応する比較条件、指標、期待方向、反証時解釈を割り当てる。
- パラメータは自由に増やさない。観測値、物理制約、既存文献、標準課題、他パラメータとの関数関係から導出できるものは従属化し、自由パラメータ台帳へ残さない。
- 文献根拠を使う場合は、実際に確認した文献だけを直前主張へ Markdown 脚注で付ける。文献リストだけに存在する引用を置かない。
- 数値算定、指標集計、表の整合確認、パラメータ感度の簡易計算には Python を使う。暗算や印象で数値主張を作らない。
- 自己弁論、自己防衛、手順説明を避ける。「指示に従う」ではなく、設計、判断、結論、実験条件を出力する。

## 入力正規化

入力から次を抽出し、欠落は `未指定` と明記する。

| 欄 | 抽出内容 |
|---|---|
| 応用場面 | 現実の用途、運用制約、失敗が問題になる状況 |
| 対象システム | エージェント、モデル、ロボット、人間、組織、データ処理系 |
| 変化条件 | 環境変動、タスク分布、資源量、通信、制約、ノイズ |
| 既存手法 | 比較対象、標準手法、暗黙の運用規則 |
| 失敗例 | 何が、どの条件で、どの観測量として悪化するか |
| 理論枠組み | 使えそうな理論、ただし根拠未確認なら候補扱い |
| 標準課題 | 実験可能な benchmark、simulation、dataset、task family |
| 観測可能 cue | 主体が局所的に観測できる物理量、構造量、密度、履歴、環境記憶 |
| 観測不能情報 | 目的対象、全体地図、最終形状、グローバル目的関数など、主体へ与えない情報 |
| 目的機能 | 温度安定、群れ形成、内部環境維持、通路形成、構造獲得など、外部評価で測る機能 |
| 既存システム・データ | シミュレータ、ログ、CT、実験データ、既存 repo、文献モデル |

## 設計プロトコル

### 1. 標準問題へ写像する

応用語をそのまま保持せず、抽象問題へ変換する。

| 要素 | 定義するもの |
|---|---|
| Actors | 意思決定主体、観測能力、行動能力 |
| Resources | 消費、共有、競合、蓄積される対象 |
| State | 環境、主体、履歴、制約の状態変数 |
| Actions | 介入可能な操作、選択、移動、割当、更新 |
| Constraints | 時間、通信、容量、物理、制度、計算量 |
| Objective | 成功条件、損失関数、達成閾値 |
| Failure boundary | 既存手法が崩れる条件 |

### 1.5. 局所 cue 閉ループへ写像する

局所観測や stigmergy を扱う場合は、対象現象を実生物の模倣としてではなく、次の因果列として再定義する。

```text
local cue
-> target selection / action selection
-> local action
-> structure or environmental memory
-> physical or informational field
-> target function
```

この列のどこを提案し、どこを評価し、どこは境界条件として固定するのかを分ける。成功例の外観類似だけを主評価にしない。

### 2. 失敗モードを分類する

既存手法の失敗を症状ではなく機構として分類する。

| 類型 | 意味 | 問うべき観測 |
|---|---|---|
| 過剰 | 必要以上に使う、集まる、反応する | 過使用率、重複、渋滞、過干渉 |
| 不足 | 必要量へ届かない | 未達率、欠損、未探索、未割当 |
| 遅延 | 反応が遅い、更新が遅い | latency、追従誤差、収束時間 |
| 偏り | 一部へ集中する | 分布歪度、負荷偏在、選択偏差 |
| 衝突 | 主体間の選択が干渉する | conflict count、collision、競合失敗 |
| 不安定 | 小変化で挙動が揺れる | 分散、振動、再現性、seed sensitivity |
| 非効率 | 成果に対して消費が大きい | コスト、エネルギー、計算量、試行数 |
| 汎化失敗 | 条件変更で性能が落ちる | out-of-distribution gap、robustness gap |

出力は `既存手法の仮定 -> 破綻条件 -> 行動症状 -> 測定可能な損失` の因果列で書く。

### 3. 提案機構を合成する

提案を「新しい名前」ではなく、失敗因果列への最小介入として定義する。

| 欄 | 内容 |
|---|---|
| Mechanism P | 提案機構の1文定義 |
| Target variable | 直接変える内部量、構造量、制約量 |
| Intervention point | 観測、判断、行動、相互作用、更新、評価のどこへ入るか |
| Operation | 発行、抑制、割当、選択、共有、記憶、忘却、再重み付けなど |
| Boundary condition | どの条件で効き、どの条件では効かないか |
| Degenerate case | 提案が既存手法や単純 heuristic へ退化する条件 |
| Failure handling | 機構が誤作動した場合の安全側挙動 |

媒介変数 `M` は必須ではない。失敗因果列を1つの状態量、権利、記憶、信号、信用、資源、制約として表すと検証可能性が上がる場合だけ導入する。

### 4. システムを分解する

実装可能な粒度で構成要素を定義する。

| 要素 | 必須項目 |
|---|---|
| 状態 | 記号、意味、観測可否、初期値、更新条件 |
| 観測 | 観測範囲、ノイズ、遅延、欠損 |
| 行動 | 実装名、意味、形成される効果、成功条件、失敗条件 |
| 相互作用 | 競合、協調、通信、共有、排他、同期 |
| 出力 | 研究主張へ接続する最終観測 |
| ログ | 再現に必要な最小記録単位 |

行動集合は「実装できる粒度」と「役割や効果が観測できる粒度」の交点で切る。
局所 cue 研究では、環境設定とエージェント設定を別々の表で固定する。環境側は断面・空間、境界条件、初期構造、外乱、目的領域、環境アブレーション、初期構造アブレーション、目的領域の位置感度を含める。エージェント側は観測可能 cue、観測不能情報、行動、action selection、target selection、環境記憶の更新式、cue 正規化、単独 cue、複合 cue、leave-one-out 条件を含める。

### 5. 中心仮説を作る

汎用形を使う。

```text
提案機構 P は、[内部判断または表現] を変化させ、[構造または失敗モード] を形成・抑制し、[対象性能] を改善する。
```

派生形は条件付きで使う。

```text
媒介変数 M による [情報/資源/権利/記憶] の制御は、[判断規則] を変化させ、[役割/分布/相互作用構造] を形成し、[集団性能/安定性/効率] を改善する。
```

### 6. 観測予測へ分解する

| 主張層 | 予測 | 例 |
|---|---|---|
| 内部変化 | 判断、表現、割当、更新則が変わる | 選択確率、閾値、重み、policy entropy |
| 構造変化 | 分布、役割、ネットワーク、軌跡が変わる | modularity、role persistence、load balance |
| 性能変化 | 目的関数が改善する | success rate、cost、completion time |
| 副作用 | 悪化や trade-off が現れる | overhead、fairness loss、fragility |

各予測は、どの比較条件で差が出れば支持され、どの結果なら棄却されるかまで書く。

### 7. 比較条件を設計する

条件名ではなく、反証役割で設計する。

| 条件クラス | 目的 | 反証する代替説明 |
|---|---|---|
| NullCondition | 機構なしの自然挙動を測る | そもそも問題が発生しない |
| ExistingBaseline | 既存手法との差を測る | 既存手法で十分 |
| SimpleHeuristic | 単純規則との差を測る | 手作り規則で十分 |
| PartialMechanism | 一部だけの効果を測る | 部品単体で十分 |
| KeyAblation | 重要部品の必要性を測る | key variable/action は不要 |
| FullProposal | 完全提案の総合効果を測る | 統合しても改善しない |

具体名は対象研究へ合わせて instantiate する。`FullProposal` だけの実験を主実験にしない。
局所 cue 研究では、all-cue 条件、single-cue 条件、composite-cue 条件、leave-one-out 条件、環境アブレーション、初期構造アブレーション、目的領域位置の感度分析を、主張へ対応する範囲で置く。

### 8. 指標を主張へ対応させる

`claim-metric matrix` を作る。

| Claim | Observable | Metric | Comparison | Expected direction | Failure interpretation |
|---|---|---|---|---|---|
| 内部変化 | 何を見るか | どう測るか | どこと比べるか | 増減または形状 | 外れた時に何が否定されるか |
| 構造形成 | 何を見るか | どう測るか | どこと比べるか | 増減または形状 | 外れた時に何が否定されるか |
| 性能改善 | 何を見るか | どう測るか | どこと比べるか | 増減または形状 | 外れた時に何が否定されるか |
| 副作用 | 何を見るか | どう測るか | どこと比べるか | 許容範囲 | 許容不能条件 |

1つの指標で複数の主張を証明しない。性能指標だけで機構主張を支えない。
局所 cue 研究では評価を `機能評価`、`行動評価`、`構造評価`、`物理場・情報場評価` の4層に分ける。外観の類似性は補助的診断に留め、主評価にしない。

### 9. 固定パラメータを列挙する

主実験前に固定すべき項目を表にする。

| 層 | 固定するもの |
|---|---|
| Environment | 地形、分布、外乱、課題系列、初期条件 |
| Agent/System | 個体数、能力、観測範囲、行動集合、モデル容量 |
| Resource/Data | 発生量、消費規則、データ分割、欠損、ラベル条件 |
| Constraint | 通信、移動、時間、電池、容量、計算量 |
| Mechanism | 変数範囲、更新頻度、共有範囲、閾値、失効条件 |
| Learning/Adaptation | 初期値、更新関数、探索率、履歴長、停止条件 |
| Evaluation | seed 数、試行回数、統計検定、信頼区間、除外規則 |

調整用実験と主実験を分離する。主実験後に都合よくパラメータを動かさない。

### 10. 恣意性を排除する

予備実験を、主張を有利にする調整ではなく、設計自由度を減らすための同定実験として定義する。

| 項目 | 要求 |
|---|---|
| Calibration objective | 主実験の性能指標ではなく、測定安定性、到達可能範囲、破綻境界、計算可能性を同定する |
| Exclusion rule | 予備実験で見た指標を、主実験の成功主張として使わない |
| Parameter ledger | 自由パラメータ、従属パラメータ、固定定数、探索範囲を分ける |
| Reduction rule | 独立に調整可能なパラメータ数を減らす規則を明示する |
| Justification | 物理、測定、理論、標準課題、実装制約、既存文献のどれに基づくかを書く |
| Freeze point | どの時点で値、範囲、更新関数、除外規則を固定するかを書く |

パラメータ縮約は次の優先順で行う。

1. 観測値から決まる量は、実測値またはデータ統計へ固定する。
2. 物理・制度・通信・計算制約から決まる量は、外部制約へ固定する。
3. スケールに依存する量は、無次元比、正規化量、容量比、時間比、密度比へ変換する。
4. 同じ機構へ属する量は、保存則、単調性、上限下限、合計一定、確率正規化で結び、独立調整を禁止する。
5. 既存手法との公平性に関わる量は、baseline matching で合わせる。
6. なお残る量だけを自由パラメータとし、探索範囲、刻み幅、選択理由を事前に固定する。

出力には `arbitrariness-control matrix` を含める。

| Parameter | Role | Status | Determination rule | Reason | Used in main claim |
|---|---|---|---|---|---|
| 記号 | 何を制御するか | fixed / derived / free / excluded | 固定値、関数、範囲、除外 | なぜその規則が妥当か | yes / no |

自由パラメータが多い場合は、提案機構の新規性ではなく調整自由度で性能差が出る危険を明記する。主張に必須でないパラメータは除外し、必須だが任意性が残るパラメータは感度分析へ回す。

### 11. 貢献を3層で書く

| 層 | 書く内容 | 避ける内容 |
|---|---|---|
| 現象 | 既存手法がどの条件で破綻するか | 単なる性能差 |
| 機構 | 破綻をどの介入で制御するか | 名前だけの新規性 |
| 評価 | 何を測ると機構を検証できるか | 汎用指標の羅列 |

「初めて」「革新的」は根拠がない限り使わない。差分、条件、反証可能性で貢献を書く。

## 条件付き拡張

### Local cue / stigmergy / distributed control

局所 cue、環境記憶、群れ、構造形成、分散制御、目的対象を直接観測しない主体を扱う場合は、`references/local-cue-closed-loop-research.md` を読む。この参照は、面白さの核、閉ループ定式化、社会意義、学術意義、目的、RQ、環境設定、エージェント設定、4層評価、新規性、文献根拠、説明文の出力契約を定義する。

### Adaptive feedback

履歴、経験、記憶、学習、オンライン更新を含む場合だけ追加する。

- 情報単位: 何が1単位の経験、観測、記憶か。
- 記憶状態: 保存場所、容量、寿命、更新頻度。
- 更新関数: 入力、出力、損失、安定化項、停止条件。
- 共有条件: いつ、誰が、どこへ、何を共有するか。
- 忘却条件: 古い情報、誤情報、外れ値をどう失効させるか。
- 失敗時挙動: 更新停止、保守的 fallback、履歴隔離。

### Multi-agent role formation

複数主体、群ロボット、分散エージェント、役割分化を扱う場合だけ追加する。

- 役割を個体属性ではなく、行動分布、位置、資源接触、通信、貢献の観測パターンとして定義する。
- 個体固定性と役割流動性を分ける。
- 構造指標は、役割持続時間、分業度、負荷分散、相互作用ネットワーク、衝突率を候補にする。
- 性能改善だけで役割形成を主張しない。

### Mediated resource allocation

資源、権利、トークン、信用、優先度、注意、通信帯域を媒介する場合だけ追加する。

- 発行: いつ、誰が、どの条件で生成するか。
- 移転: 主体間で移るか、環境へ沈着するか、中央管理されるか。
- 消費: 何を実行すると減るか。
- 枯渇: 不足時の制御則。
- 過剰: 蓄積、独占、投機、渋滞の抑制。
- 競合: 同時要求、衝突、優先順位、ロールバック。

## 出力形式

指定がなければ次の順で出力する。

1. 研究タイトル
2. 一文説明
3. 背景
4. 既存手法の限界
5. 提案機構
6. 数理モデル
7. 実装手順
8. 比較条件
9. 評価指標
10. 予備実験と恣意性排除設計
11. 中心仮説
12. 論文としての貢献
13. 未確認事項と次の検証

各節は「問い -> 判断 -> 理由 -> 次の問い」で接続する。概要から入り、抽象から具体へ落とす。用語、数式、条件は必要になった段階でだけ導入する。

## 文章規律

- 学術的、圧縮的、名詞句中心の表現を使う。
- 断定できない箇所は `確認済み`, `推測`, `未確認` に分ける。
- 「可能性があります」を濫用せず、条件付き命題として書く。
- 「有用である」ではなく、どの条件で、何を、どの比較に対して改善するかを書く。
- 「興味深い」「重要」だけで価値を述べない。機構差、観測差、評価差で述べる。

## スライド生成時の様式契約

研究スライド、スライド画像、発表資料を生成する場合だけ適用する。

- 白背景、黒を主論理、グレーを区画、色は系列識別に限定する。
- 重要語は黒太字と下線で示す。結論は黒背景に白文字で置く。
- グラデーション、過度な配色、余白過多、アイコン中心のビジネス資料風を避ける。
- 各パネルは `黒太字+下線の見出し -> 図表または模式図 -> 短い解釈文` の順にする。
- 視線の流れは `背景 -> 目的 -> 方法 -> 結果 -> 考察 -> 意義/今後` とする。
- 文字サイズは title 40pt、section 26pt、panel 21pt、body 16pt、caption 12pt を基準にする。
- 高密度化は小さい文字ではなく、図表化、模式図化、短文化で行う。

## 品質ゲート
