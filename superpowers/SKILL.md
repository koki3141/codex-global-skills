# Superpowers for Codex

## 目的

非自明なソフトウェア変更を、思いつきの実装ではなく次の品質ゲートで進める。

`目的確認 → 現状把握 → 設計 → 実装計画 → 隔離作業 → TDD → 仕様適合レビュー → コード品質レビュー → 最終検証`

これは `obra/superpowers` v6.1.1 の中核原則を、現在のグローバルCodex skill構成へ統合したローカル互換プロファイルである。公式pluginそのものではなく、SessionStart hookや自動強制起動には依存しない。

## 起動ゲート

次のいずれかを満たす場合に使う。

- ユーザーが `$superpowers` を明示した。
- 複数ファイルまたは複数コンポーネントを変更し、失敗時の手戻りが大きい。
- 仕様、設計、実装、テスト、レビューを分離して監査可能にしたい。
- 実装前に設計承認、または実装後に独立レビューが必要である。

次では起動しない。

- 小さく可逆な修正。
- 既知のコマンド一つで完了する作業。
- コード変更を伴わない単純な調査や説明。
- ユーザーが明示的に最短経路を優先している作業。

## 優先順位

1. 最も近い `AGENTS.md`、プロジェクト規則、ユーザー指示。
2. 安全性、権限、秘密情報、不可逆操作の制約。
3. このskillの品質ゲート。
4. 実装対象に固有のdomain skill。

既存の厳しいローカル規則を弱めてはならない。

## 標準フロー

### 1. 目的と完了条件を固定する

曖昧な依頼では `$define-goal` を使い、対象、非目標、受入条件、制約、許可された副作用を短く記録する。完全に仕様化済みなら質問を増やさず、その前提を明示して進む。

### 2. 現状とベースラインを確認する

実装前に、該当する `AGENTS.md`、README、設計資料、既存テスト、依存関係、`git status` を確認する。バグ調査では `$evidence-gated-investigation` を先に使い、原因仮説を実装と混同しない。

### 3. 設計を決める

重要なAPI、データ契約、状態遷移、永続化、セキュリティ、移行方式を変更する場合は、候補、選定理由、棄却案、リスク、検証方法を提示する。設計判断が重大かつ対立的なら、実装前に `$agent-arena` へエスカレートする。

### 4. 実装計画を作る

タスクを小さな完了単位へ分解し、各タスクに対象ファイル、変更内容、テスト、完了条件、依存関係を付ける。詳細は `references/workflow.md` を読む。

### 5. 作業を隔離する

原則としてmain/masterへ直接実装しない。プロジェクト規則に従い、topic branchまたはworktreeを使う。既存の未コミット変更を上書き、stash、reset、cleanしてはならない。

### 6. TDDで実装する

観測可能な振る舞いには、失敗するテストを先に作り、失敗を確認してから最小実装を行い、成功後にリファクタリングする。テスト不可能な変更では、代替の決定的検証と理由を記録する。

### 7. 二段階レビューを通す

各まとまりについて、最初に仕様適合、次にコード品質を別の観点で確認する。作成者の自己レビューだけで代替しない。Codexの `/review`、read-only subagent、`$code-review-excellence` を使い分ける。詳細は `references/review-gates.md` を読む。

### 8. 最終検証する

`$verification-loop` を使い、対象テスト、lint、type check、build、統合テスト、最終diff、文書、互換性、移行・ロールバック条件を確認する。未実行の検証を成功扱いしない。

### 9. 完了を判定する

重大な主張は `$claim-calibration` で、確認済み、推定、未確認を分離する。未解決のImportant/Critical指摘を残したまま完了としない。push、merge、deploy、公開はユーザー指示とプロジェクト規則に従う。

## サブエージェント

並列化は既定値ではない。独立した非自明な作業が複数あり、所有権と検証経路を分けられる場合だけ `$cost-aware-subagents` を使う。

- 読み取り、探索、レビューは原則read-only。
- 同じ論理的write setへ複数writerを置かない。
- 統合、最終diff、最終検証は親が行う。
- 仕様レビューと品質レビューは、可能なら実装者と別コンテキストにする。
- 同系列モデルだけの一致を独立証拠とみなさない。

## Agent Arenaへのエスカレーション

`$agent-arena` は次の場合だけ追加する。

- 不可逆または高blast-radiusの設計判断。
- 複数の妥当な案があり、一次資料や再現で争点を裁定できる。
- 重大なroot cause仮説が競合している。
- セキュリティ、データ損失、移行、公開API、研究結論などで単一モデルの過信が危険である。

日常的な小変更、通常のdiffレビュー、決定的テストで解ける問題に討論を追加しない。

## 完了報告

最低限、次を報告する。

- 採用した設計と重要な前提。
- 変更ファイルまたはコミット。
- 実行した検証と結果。
- 仕様レビューと品質レビューで受理・却下した指摘。
- 残存リスクと未確認事項。
- Agent Arenaまたはサブエージェントを使った場合、その構成と劣化条件。
