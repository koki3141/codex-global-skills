# Skill Improver

Execute improvement plans generated by `skill-quality-reviewer` to automatically update and fix issues in Claude Skills.

## Core Workflow

### Plan-Based Mode

```
Read improvement-plan-{name}.md
    ↓
Parse improvement items (High/Medium/Low priority)
    ↓
Group changes by file
    ↓
Detect and resolve conflicts
    ↓
Backup original files
    ↓
Execute updates (Edit or Write tools)
    ↓
Verify results
    ↓
Generate update-report
```

### Conversation-Derived Mode

Use this mode when the user asks to improve or evolve skills from the current conversation, prior corrections, or observed repeated failures, and no `improvement-plan-*.md` is provided.

```
Read named skill(s)
    ↓
Extract repeated corrections and durable preferences from the conversation
    ↓
Convert them into concrete trigger rules, style rules, and verification checks
    ↓
Choose the smallest affected skill set
    ↓
Backup original files
    ↓
Patch skill instructions
    ↓
Verify YAML, references, and trigger coverage
    ↓
Generate or summarize update report
```

## Step-by-Step Guide

### Step 1: Load the Improvement Plan or Derive One

Read the `improvement-plan-{skill-name}.md` file generated by `skill-quality-reviewer`.

```bash
# Plan is typically in current directory
ls improvement-plan-*.md

# Or specify full path
read /path/to/improvement-plan-my-skill.md
```

**Validate the plan:**
- File exists and is readable
- Contains priority sections (High/Medium/Low)
- Has structured improvement items
- Includes file paths and suggested changes

See `references/plan-format.md` for detailed plan structure.

**If no improvement plan exists:**
- Identify the skill(s) explicitly named by the user.
- If the user asks for conversation-derived improvements, treat repeated corrections, rejected wording, and successful final phrasing as the improvement source.
- Convert each durable correction into a rule with:
  - Trigger or applicability condition
  - Bad pattern to avoid
  - Preferred behavior
  - Verification method, usually an `rg` search, schema check, or example phrase check
- Do not update broad adjacent skills merely because they seem related. Patch the smallest skill set that would prevent the observed failure.
- If the target skill is unclear, inspect recently used skills and update only the one whose instructions failed to prevent the repeated issue.

### Step 2: Parse and Group Changes

Extract all improvement suggestions and organize by target file.

**Extract from each item:**
- File path (e.g., `SKILL.md:line:line` or `references/plan-format.md`)
- Dimension (Description Quality, Content Organization, etc.)
- Impact (+X points)
- Current content
- Suggested content
- Reason for change

For conversation-derived updates, extract:
- User correction
- Failing output pattern
- Durable rule to add
- Target section in `SKILL.md`
- Regression search or manual check

**Build update queue:**
```
by_file = {
  "SKILL.md": [change1, change2, ...],
  "references/plan-format.md": [change3, ...],
  "examples/demo.md": [change4, ...],
}
```

### Step 3: Detect and Resolve Conflicts

Check for conflicts when multiple changes affect the same content.

**Resolution strategy:**
1. High priority takes precedence over medium/low
2. If same priority, preserve first change
3. Flag conflicts for manual review
4. Document resolution in update report

See `references/merge-strategies.md` for detailed merge logic.

### Step 4: Sort by Priority

Order changes by priority within each file.

**Priority order:**
1. High Priority (execute first)
2. Medium Priority (execute second)
3. Low Priority (execute last)

### Step 5: Backup and Execute

**Backup location:** `~/.codex/skills/backup/{skill-name}-{timestamp}/`
Backups must be restorable snapshots, not active skills. Store the copied
entrypoint as `SKILL.snapshot.md` inside `backup/` so historical versions do not
appear in Codex skill discovery.

```bash
# Use backup script
~/.codex/skills/skill-improver/scripts/backup-skill.sh <skill-path>
```

**Apply changes:**
- Use Edit tool for existing content
- Use Write tool for new files
- Verify each change was applied

### Step 6: Verify and Report

**Verification checks:**
1. YAML syntax valid
2. All modified files exist and are valid
3. New files were created successfully
4. No unintended changes occurred
5. New trigger phrases or rules cover the user’s actual request
6. The updated skill does not conflict with existing higher-priority instructions

```bash
# Use verify script
~/.codex/skills/skill-improver/scripts/verify-update.sh <skill-path>
```

**Generate update-report-{skill-name}-{timestamp}.md** documenting:
- Summary (files modified, files created, total changes)
- Changes Applied (per-file breakdown)
- Quality Improvement (before/after scores)
- Verification Results
- Backup Location

See `examples/update-report-example.md` for report template.

## Priority Handling

### High Priority
Execute first. These typically address:
- Critical description issues
- Major writing style problems
- Missing structural elements
- Security-related concerns

### Medium Priority
Execute after High. These typically address:
- Content organization improvements
- Additional examples
- Documentation enhancements

### Low Priority
Execute last. These typically address:
- Minor clarifications
- Nice-to-have improvements
- Polish and refinement

## Best Practices

### Before Applying Updates

1. **Review the improvement plan** - Understand what will change
2. **Verify backup location** - Ensure backups can be restored
3. **Check for manual changes** - Note any uncommitted local modifications
4. **Estimate impact** - Review expected quality improvement
5. **For conversation-derived updates, separate durable behavior from one-off task content** - Add reusable rules, not the specific answer text from the conversation.

### During Update Execution

1. **Process in priority order** - High → Medium → Low
2. **Verify after each file** - Check updates were applied correctly
3. **Log all changes** - Document what was modified
4. **Handle conflicts gracefully** - Flag for review if needed

### After Applying Updates

1. **Review the update report** - Confirm all changes were intended
2. **Test the skill** - Verify it still works correctly
3. **Compare scores** - Check quality improvement matches expectations
4. **Keep backup** - Retain backup until confident in changes
5. **Run regression searches** - Search for the exact bad patterns that triggered the update when the skill edits involve wording or formatting.
6. **Check discovery hygiene** - Ensure no `backup/**/SKILL.md` files remain unless the user explicitly wants backup snapshots to be callable skills.
